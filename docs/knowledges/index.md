# 知识库索引

> 本文件是 docs/knowledges/ 的唯一入口。
> **所有新增知识文件必须在此登记**，否则视为未沉淀。
>
> **Agent 使用规则**：在每个阶段开始前，必须检查本索引中是否有与当前任务相关的条目。
> 检查方式：按任务类型、功能领域、技术栈 三个维度匹配。

## 目录结构规范

```
docs/knowledges/
├── index.md                    ← 本文件（必须保持最新）
├── domain/                     ← 业务领域知识（术语表、业务规则）
│   └── {domain}-glossary.md
├── architecture/               ← 架构决策记录（ADR）
│   └── {NNN}-{decision}.md
├── patterns/                   ← 可复用代码/设计模式
│   └── {pattern-name}.md
├── standards/                  ← 编码规范（按技术栈）
│   └── {tech}-standards.md
├── lessons-learned/            ← 经验教训（踩坑记录）
│   └── {NNN}-{lesson-title}.md
└── guides/                     ← 操作指南
    └── {guide-name}.md
```

---

## 知识文件格式要求

每个知识文件必须包含以下 Meta 区域：

```markdown
## Meta
- **类型**：经验教训 | 设计模式 | 编码规范 | 架构决策 | 操作指南
- **适用场景**：什么情况下应该查阅本文件
- **关键词**：逗号分隔的检索关键词（供 Agent 匹配使用）
- **创建时间**：YYYY-MM-DD
- **来源功能**：产生此知识的功能名（如适用）
```

## 知识消费协议

Agent 在以下时机必须检索本索引：

| 时机 | 检索维度 | 目的 |
|------|---------|------|
| P1 需求分析开始前 | 业务领域关键词 | 避免重复定义已有术语，识别已知约束 |
| P3 技术设计开始前 | 技术栈 + 功能类型 | 复用已有模式，避免重复踩坑 |
| P4 编码开始前 | 技术栈 + 模块名 | 遵循已沉淀的编码规范 |
| bug-fix P3 开始前 | Bug 类型 + 受影响模块 | 检查是否有相同根因的历史修复经验 |

**检索步骤**：
1. 读取本索引文件
2. 按关键词列匹配当前任务相关条目
3. 对匹配条目，读取文件内容
4. 在执行输出中注明"参考了知识条目 {路径}"

---

## 知识归档协议

P5 GC 阶段，如有新知识需要沉淀：
1. 按目录结构选择正确的子目录
2. 使用标准 Meta 格式创建文件
3. **必须在本索引表追加一行**（漏登记 = 知识丢失）
4. 更新 `workspace/memory.md §B 经验教训` 中的摘要（如果是重要教训）
