# 存量项目接入 AIFD 框架指南

> 适用场景：项目已有代码和业务逻辑，希望引入 AIFD 框架改善研发质量和文档体系。

---

## 核心原则：渐进而非革命

**不要试图一次性迁移全部代码。**

正确的做法是：
1. 框架与现有代码**共存**
2. 新功能**完全走 AIFD 流程**
3. 旧代码**择机、按需**补充文档和改善分层
4. 通过 GC 循环逐步提升整体质量

---

## 第一阶段：框架初始化（1-2 小时）

### 步骤 1：运行 `/init-project`

`/init-project` 会引导完成以下配置：
- 填写 `CLAUDE.md` 项目定制区
- 选择技术栈 Agent
- 初始化 `workspace/stage-status.json`
- 配置 hook 守卫路径

**存量项目注意**：在问题 4（项目现状）中选择"存量项目"，`/init-project` 会启用额外的现状梳理流程。

### 步骤 2：梳理现有架构

使用 `arch-agent`（模式 A）对现有代码库进行快速梳理，产出：

**文件：`docs/knowledges/architecture/000-initial-state.md`**

内容包含：
- 系统当前的主要模块列表
- 识别到的分层违规（不符合 AIFD 四条核心原则的地方）
- 现有 API 端点摘要（不需要完整文档，只要有一个列表）
- 主要技术债清单

**不需要：** 完整的技术文档，只需要一个"现状快照"。

### 步骤 3：建立核心追溯起点

为最核心的 2-3 个模块创建**简化版**技术文档（不走完整 P1-P3 流程）：

```
docs/specs/legacy-{module-name}/
  └── tech.md  （简化版，只记录接口清单和数据模型）
```

这些文档为后续新功能开发提供"追溯起点"，不需要需求文档和产品设计文档。

---

## 第二阶段：新功能用新流程（立即开始）

### 规则：从今天起，新功能全走 AIFD 流程

哪怕项目中 90% 是遗留代码，新功能也必须：
- 走完整 P1 → P2 → P3 → P4 → P5 流程
- 产出完整的 requirement.md / tech.md / exec-plan.md
- Code Review 中如果发现新代码依赖了遗留代码的分层违规部分，记录为技术债

**实践建议**：新功能尽量"隔离"，避免深度依赖遗留代码的违规部分。

---

## 第三阶段：渐进改善旧代码（按需执行）

### 何时触发旧代码改善

以下场景触发对应旧代码的补文档或重构：
- **需求变更**：旧功能需要新增/修改时，同步补写该功能的 tech.md
- **Bug 修复**：修复旧代码 Bug 时，走 `/bug-fix` 流程（会自动要求写技术分析）
- **重构决策**：当旧代码的技术债严重影响开发效率时，走 `/new-feature` 流程的重构路径

### 补写文档的策略（降级版）

旧功能补写文档时，不需要走完整流程：

| 场景 | 最小文档要求 |
|------|------------|
| 功能从未变化、只是补文档 | 只写 tech.md + traces_to |
| 功能需要小修改 | tech.md + exec-plan（里程碑粒度可以更粗） |
| 功能需要较大变更 | 走完整 P3 → P4 流程 |
| 功能需要重新设计 | 走完整 P1 → P5 流程 |

### 分层改善的优先级

优先改善以下类型的违规（收益最高）：

1. **数据层 import 业务层**：通常是为了规避循环依赖，但会导致层次混乱 → 提取接口
2. **框架类型泄露到业务层**：如 Service 层直接使用 `HttpRequest` → 在边界层转换为 DTO
3. **全局变量/单例状态**：难以测试，难以追溯 → 改为依赖注入
4. **巨型文件（>600行）**：理解难度高 → 按领域拆分

## 第四阶段：追踪迁移进度

### 在 `docs/knowledges/architecture/000-initial-state.md` 中追踪

添加迁移进度区域：

```markdown
## 迁移进度追踪

| 模块 | 文档状态 | 分层合规 | 上次更新 |
|------|---------|---------|---------|
| user-service | ✅ 已有 tech.md | ⚠️ 部分违规 | 2026-03 |
| order-service | ❌ 无文档 | ❌ 多处违规 | — |
| payment-service | ✅ 已有 tech.md | ✅ 合规 | 2026-04 |
```

### GC 阶段的迁移检查项

每次 GC 循环时，额外检查：
- [ ] 本次修改是否顺带改善了相关旧代码的分层违规？
- [ ] `000-initial-state.md` 的迁移进度是否需要更新？
- [ ] 技术债清单中的高优先级项是否可以安排下一轮改善？

## 常见问题

**Q：遗留代码不满足 hook 守卫要求怎么办？**

A：hook 守卫只在 **没有活跃实现计划** 时拦截。只要有 `stage-status.json` 中有 `stage=implementation, status=running` 的功能，就可以编辑任何代码路径。修改旧代码时，先创建一个轻量级的执行计划（哪怕只有1-2个里程碑），然后正常进行。

**Q：旧代码没有测试，QA Agent 执行阶段怎么办？**

A：在 exec-plan 中明确标注"本里程碑补充遗留代码的测试覆盖"。测试是代价，但不是障碍——先写测试确认现有行为，再改代码。

**Q：遗留项目的 `docs/specs/index.md` 初始状态是什么？**

A：留空，只记录从引入 AIFD 框架之后新开发的功能。历史功能用 `000-initial-state.md` 中的模块列表代替，不需要全量补录 index.md。

**Q：多少比例的旧代码"合规"后才算迁移成功？**

A：没有硬性指标。实用的判断标准：**新功能开发时不再被旧代码的违规拖累**。当你发现新功能可以在隔离的分层内干净实现，说明迁移到位了。
