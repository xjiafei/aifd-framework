# 质量标准、能力边界与指标

---

## 质量保证的诚实边界

### 框架能保证

| 保证项 | 说明 |
|--------|------|
| **不跳步** | 每个阶段必须完成才能进入下一阶段，流程引擎强制执行 |
| **不漏项** | Checklist 驱动，所有必检项逐一验证 |
| **结构性预检** | Subagent 在人类审查前自动捕获格式、结构、完整性问题 |
| **实现完整性（Close-Loop）** | 每个需求条目必须可追踪到实现和验证，不允许"遗留处理" |
| **可追溯** | 需求 → 设计 → 实现 → 验证，全链路通过 `traces_to` 关联 |
| **状态可恢复** | 任何中断后可从 `stage-status.json` 恢复到断点继续执行 |

### 框架不能保证

| 局限 | 说明 |
|------|------|
| **领域正确性** | 业务逻辑是否符合真实业务规则，依赖人类判断 |
| **零熵增** | 随时间推移，文档与代码可能出现不一致，需要 GC 循环对抗 |
| **需求质量** | 输入的需求本身是否合理、完整，框架无法判断 |
| **无限循环** | Agent 可能在验证-修复循环中反复，需设置最大轮次上限 |

## 两级审查机制

```
代码/文档变更
     │
     ▼
┌─────────────┐    捕获结构性/格式问题
│ Subagent 预检 │──→ 自动检查 Checklist 合规性
└──────┬──────┘    格式校验、依赖方向、命名规范
       │
       ▼ 预检通过
┌─────────────┐    把关语义方向和业务正确性
│  人类终审    │──→ 审查业务逻辑、架构决策
└─────────────┘    确认是否符合项目目标
```

- **Subagent 预检**：自动执行，无需人工参与。检查项包括：文件格式、Checklist 完整性、依赖方向合规、错误信封一致性。
- **人类终审**：Subagent 预检通过后提交人类。人类关注：业务正确性、架构合理性、需求覆盖度。

## 执行自治等级（L0 - L3）

| 等级 | 名称 | 行为范围 | 人类参与 |
|------|------|---------|---------|
| **L0** | 辅助态 | 解释代码、搜索信息、起草文档 | 全程人类驱动 |
| **L1** | 受限改动 | 单文件内改动（修 bug、加注释、调格式） | 每次改动必须人工 review |
| **L2** | 有计划局部自治 | 先写执行计划（plan），在 bounded context 内实现，验证后提交人工 review | 计划需人工批准，实现后人工 review |
| **L3** | 端到端修复 | 高测试覆盖区域，从问题复现到提交 PR 一路完成 | 仅 PR 层面 review |

> **当前默认等级：L2**
>
> Agent 必须先产出执行计划并获得批准，然后在计划范围内自主实现，完成后提交人类审查。

---

## 需求自治等级（R0 - R2）

| 等级 | 名称 | 行为 |
|------|------|------|
| **R0** | 人定义任务 | 人类明确定义需要做什么，Agent 按指令执行 |
| **R1** | Agent 建议人批准 | Agent 主动识别改进点并提出建议，人类决定是否采纳 |
| **R2** | 策略内自主发起 | Agent 在预设策略范围内自主发起任务（如自动修复 lint 错误） |

> **当前默认等级：R0**
>
> 所有任务由人类发起和定义，Agent 不自主创建新任务。

## 诊断层指标

> 目标：发现系统哪里在变坏。

| 指标 | 含义 | 告警阈值（建议） |
|------|------|----------------|
| **循环依赖数** | 模块间循环引用的数量 | > 0 即告警 |
| **跨层违规数** | 违反分层依赖方向规则的 import 数量 | > 0 即告警 |
| **术语漂移** | 同一概念在不同文档/代码中使用不同命名的实例数 | > 3 需清理 |
| **错误 envelope 变体数** | 偏离统一错误信封结构的 API 响应数 | > 0 即修复 |
| **Stale docs 数** | 与代码实际状态不一致的文档数量 | > 2 触发 GC |
| **E2E 覆盖率** | E2E 用例覆盖的关键用户路径数 / product.md 中定义的关键路径总数 | < 80% 需补充 |
| **E2E 脚本覆盖** | Prong B 生成的 .spec.ts 脚本数 / 总 E2E 用例数 | < 50% 需评估 CI 回归风险 |

## 运营层指标

> 目标：衡量框架运行效率。

| 指标 | 含义 | 计算方式 |
|------|------|---------|
| **Plan Coverage** | 执行计划覆盖需求条目的比例 | 计划中涉及的需求条目数 / 总需求条目数 |
| **Invariant Coverage** | 架构不变量被检查覆盖的比例 | 已验证不变量数 / 声明的不变量总数 |
| **Validation Coverage** | 实现完成后通过验证的比例 | 验证通过的条目数 / 总实现条目数 |
| **Cleanup Half-life** | 发现问题到修复完成的中位耗时 | 统计历史修复周期的中位数 |
| **Human Attention Minutes per Change** | 每次变更需要人类关注的分钟数 | 人类实际 review 时间 / 变更次数 |
| **E2E Pass Rate** | E2E 测试通过率 | E2E PASS 数 / E2E 总执行数 |

## GC 循环

### 每次全流程完成后

完成一个完整的需求交付流程后，执行以下检查：

- [ ] `architecture.md` 是否需要更新？（新增模块、技术栈变更、架构调整）
- [ ] 本次经验教训是否已回写到 `workspace/memory.md`？
- [ ] 是否存在 stale docs？（与实际代码不一致的文档）
- [ ] `traces_to` 链接是否全部有效？（指向的目标文件/章节是否存在）
- [ ] E2E 测试脚本是否仍可运行？（选择器、URL 是否因 UI 变更而失效）

### 每 3 个功能迭代后

每完成 3 个功能的交付，进行一次框架自省：

- [ ] 框架流程是否有冗余步骤可以简化？
- [ ] 是否有反复出现的问题需要补充新的检查项？
- [ ] Checklist 模板是否需要更新？
- [ ] 自治等级是否需要调整？（根据项目成熟度和测试覆盖率）
