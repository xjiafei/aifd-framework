## 分层架构规则

### 四条核心原则（技术栈无关）

无论使用什么技术栈，以下 4 条原则不变：

| # | 原则 | 说明 |
|---|------|------|
| **P1** | **单向依赖** | 层间依赖只允许一个方向（内层不依赖外层），禁止循环依赖 |
| **P2** | **边界校验** | 外部数据（HTTP 请求、消息队列、文件输入）必须在边界层完成校验和类型转换，不允许原始外部数据进入业务逻辑层 |
| **P3** | **业务隔离** | 业务逻辑层不直接依赖具体框架/基础设施（不 import Spring 注解、不 import ORM 类型），通过接口抽象解耦 |
| **P4** | **错误统一** | 所有错误必须包含 code + message + context 三要素，通过统一 envelope 返回 |

### 依赖方向规则

- **只允许从外向内依赖**（边界层 → 业务层 → 数据层 → 定义层）
- **禁止反向**：数据层不得引用业务层，业务层不得引用边界层
- **同层慎用互引**：避免形成同层循环依赖

```
允许: 边界适配层 → 业务逻辑层 → 数据访问层 → 数据定义层
禁止: 数据访问层 → 业务逻辑层
禁止: 业务逻辑层 → 边界适配层
```

### 横切关注点（Cross-cutting Concerns）

日志、监控、认证、错误处理等横切关注点不直接跨层调用，而是通过 **Providers 接口** 统一注入：

```
┌─────────────────────────────────┐
│         Providers 接口           │
│  (Logger / Auth / Metrics / …)  │
└─────────────────────────────────┘
        ↑ 各层通过接口访问
```

- Providers 在数据定义层或配置层定义接口，在边界层注入实现。
- 各层通过依赖注入获取 Provider，不直接 import 具体实现。

## 服务间通信

> 适用于多仓库 / 微服务架构场景。

| 通信方式 | 场景 | 协议 |
|---------|------|------|
| 同步调用 | 实时查询、需要即时响应 | HTTP REST / gRPC |
| 异步消息 | 事件通知、最终一致性 | Kafka / RabbitMQ |

### 核心原则

1. **禁止直接访问其他服务的数据库** —— 所有跨服务数据获取必须通过 API。
2. 同步调用需设置合理的超时和熔断策略。
3. 异步消息需保证幂等消费。

---

## 数据库边界

- **每个服务独占一个数据库（或 Schema）**，数据库边界 = 服务边界。
- 共享数据通过 API 暴露，不通过共享数据库表。
- 数据库迁移脚本纳入版本管理，随代码一起部署。

## 错误模型

所有 API 统一使用以下错误封装结构：

```json
{
  "success": false,
  "error": {
    "code": "ORDER_NOT_FOUND",
    "message": "订单不存在",
    "details": {
      "orderId": "12345"
    },
    "traceId": "abc-def-ghi-jkl"
  },
  "timestamp": "2026-01-01T00:00:00Z"
}
```

### 错误码规范

| 前缀 | 含义 | 示例 |
|------|------|------|
| `AUTH_` | 认证/授权 | `AUTH_TOKEN_EXPIRED` |
| `VALIDATION_` | 参数校验 | `VALIDATION_FIELD_REQUIRED` |
| `BIZ_` | 业务逻辑 | `BIZ_INSUFFICIENT_STOCK` |
| `SYS_` | 系统内部 | `SYS_DB_CONNECTION_FAILED` |
